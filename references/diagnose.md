# 被罚复盘手册

用户被限流、删除、警告或封禁后来求助时用。复盘的价值有两个：搞清这次为什么，和让下次不再犯。禁止两件事：没证据就断言原因；给出"限流N天自动恢复"这类编造的确定性结论。

## 第一步：收集证据

请用户提供（有什么给什么，别逼全）：

- 平台通知原文/截图（处罚类型、条款引用是最强证据）；
- 被处理内容的完整稿件/成品（含标题、封面、话题、评论区）；
- 发布时间、数据表现时间线（正常→异常的拐点）;
- 发布页当时的标注选择（AI/营销/转载选了什么）；
- 账号近期状况（是否连续被罚、是否新号、是否改过定位）；
- 行为层情况（是否短期大量发布/频繁删改作品、评论区导流、互刷或买量——这些属经验级信号，可能影响账号综合评估，归因时并入 D 类不单独定罪）。

## 第二步：四类归因（分开，不许混）

| 类别 | 特征 | 判断依据 |
|---|---|---|
| A. 可证实违规 | 平台通知明确引用条款 | 对照规则文档找到命中的具体规则 |
| B. 披露/标注问题 | 内容本身没事，但该标没标（AI/广告/转载/虚构） | 检查发布页选项与内容实际情况的差距 |
| C. 推荐质量问题 | 没有处罚通知，只是数据差 | 标题党、封面与内容不符、完播差——这不是违规，是内容问题 |
| D. 未知信号 | 无通知、无明显违规、数据异常 | **如实写"未知"**。不编"敏感词限流"，不编恢复天数 |

平台通知没给原因时，用规则文档对内容做一次完整审核，把发现的风险项列为"可能原因（按可能性排序）"，明确标注这是推测。

## 第三步：给出行动建议

- A 类：能申诉的给申诉要点（引用规则依据组织申诉理由）；能修改重发的给修复方案（按 repair.md）；
- B 类：下次发布的标注清单；
- C 类：如实告知这是内容优化问题，不属于合规范畴；
- D 类：建议观察 + 检查账号通知中心 + 后续内容按更稳妥尺度发。

## 第四步：沉淀（复盘的真正价值）

按此模板存入 `data/cases/YYYY-MM-DD-简述.md`：

```markdown
# 案例：<一句话描述>
- 日期/平台/处罚类型：
- 平台通知原文：
- 内容关键片段（涉及隐私可脱敏）：
- 归因：A/B/C/D + 分析
- 命中规则：<规则编号或"未知">
- 学到的教训：
- 已固化：<写进 my-rules.md 的条目，或"无可固化项">
```

可提炼的教训按性质分流进 `data/my-rules.md`（判法选择表见 SKILL.md 沉淀协议）：有明确词面 → 黑名单表（默认语境判；用户确认零容忍才用见词即报）；按意思判的教训 → 语义规则表（`MY-` 编号 + 具体锚点 + 改法，来源写本案例文件名）；纯语境背景 → 语义笔记。最后向用户复述：这次的结论是什么、沉淀了什么、下次审核会自动带上什么。
